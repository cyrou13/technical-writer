## OUTPUT LANGUAGE — STRICT

All artifacts written by this command (every file under `docs/`, item
frontmatter values such as `title`/`description`, body content,
`[TODO]`/`[GAP-...]` markers, and aggregated reports) MUST be written
in **English**, regardless of the user's conversational language or
any global `CLAUDE.md` instruction. Conversational replies to the user
MAY follow the user's language; written outputs are English-only. This
applies to every sub-agent and skill invoked from this command.

Exécute le pipeline complet de génération documentaire IEC 62304 Classe A.
Sortie locale en Markdown dans `docs/`. Ne pousse rien à l'extérieur.

## Étapes

0. **Prérequis MAP (informationnel)** : les items sous `docs/items/MAP/`
   sont des **inputs manuels** (master / stakeholder / system
   requirements upstream — recopiés du Master Plan, du PMAP, ou
   équivalent). Si `docs/items/MAP/` est vide, ce n'est pas bloquant,
   mais la traçabilité §3 du livrable d'export sera incomplète : les
   SRS générés n'auront pas de `links.parent`. Prévenir l'utilisateur
   au début que pour un livrable RAQA complet, les MAP doivent être
   saisis via `/doc-item MAP-XXX-NNN` avant `/doc-srs-export`.

1. **Cartographier** le repo en lançant le sub-agent `code-archeologist`.
   Bloquant — attendre la fin avant l'étape 2.

2. **Rédiger SRS en premier** : `requirements-writer` → `docs/items/SRS/`.
   Bloquant pour que les agents suivants disposent des IDs SRS. Si des
   items MAP existent, l'agent peut tenter de remplir `links.parent`
   en associant chaque SRS à son MAP le plus probable — sinon laisser
   vide.

3. **Rédiger en parallèle** (un seul message, plusieurs Agent calls) :
   - `architecture-writer` → `docs/items/SDS/` (remplit `links.implements`).
   - `test-evidence-collector` → `docs/items/TC/` (remplit `links.verifies`).

4. **Analyse de risques safety** : lancer `risk-analyst`. Il écrit dans
   `docs/items/RSK/`, peut créer de nouvelles SRS de mitigation
   (`priority: Must`, `links.mitigates`), et peut ajouter
   `links.mitigates` à des items SDS/TC existants. Bloquant.

5. **Analyse de risques cyber** : lancer `security-analyst`. Il écrit
   dans `docs/items/THR/`, peut créer des SRS/TC de mitigation cyber,
   peut ajouter `links.mitigates` (vers THR) à des items existants, et
   peut renseigner `links.triggers` quand une menace cyber déclenche un
   hazard safety. Bloquant.

5bis. **Analyse de risques production** : lancer `production-risk-analyst`.
   Il scanne les artefacts de production (Dockerfile, .github/workflows,
   package.json scripts, deploy/, helm/, etc.), écrit dans
   `docs/items/PRSK/`, peut créer des SRS/SDS/TC de mitigation
   production, peut renseigner `links.triggers` vers RSK ou THR. Si
   **aucun** artefact de production détecté → l'agent rapporte
   "Production phase not applicable" et on passe directement à l'étape 6.
   **Recommandé pour Classe IIb** (AAMI TIR57 + IEC 81001-5-1 §6.1).
   Bloquant.

6. **Analyse usability (IEC 62366-1)** : lancer `usability-analyst`. Il
   scanne les composants UI (s'il y en a), écrit dans `docs/items/USC/`
   et `docs/items/URSK/`, peut créer des SRS/TC de mitigation usability,
   peut ajouter `links.mitigates` (vers URSK) à des items existants, et
   peut renseigner `links.triggers` quand une use error déclenche un
   hazard safety. Si **aucune surface UI** détectée → l'agent rapporte
   "62366-1 non applicable" et on passe directement à l'étape 7.
   Bloquant.

7. **Build** : `python tools/build_docs.py`. Vérifier que tous les
   fichiers `docs/generated/{10_SRS,20_SDS,30_STD,40_traceability,50_risk_analysis,60_cyber_risk_analysis,70_usability_analysis,_to_implement}.md`
   et `coverage.json` ont été produits.

8. **Revue** : `compliance-reviewer` écrit
   `docs/generated/99_compliance_review.md`.

9. **Suggestion d'export** : si `dt-config.yaml` existe ET l'utilisateur
   a saisi au moins un item MAP, suggérer en fin de synthèse :
   « pour produire le livrable QMS-ready, lance `/doc-srs-export`. » Si
   `dt-config.yaml` ou MAP manquent, lister ces gaps comme prérequis
   au livrable RAQA (informationnel, non bloquant pour `/doc-62304`).

10. **Synthèse à l'utilisateur** (≤ 16 lignes) :
   - métriques de couverture (impl + vérif) depuis `coverage.json`,
   - RSK ouverts / RSK avec résiduel non acceptable (alerte si > 0),
   - THR ouverts / THR avec résiduel non acceptable (alerte si > 0),
   - URSK ouverts / URSK avec résiduel non acceptable (alerte si > 0),
   - nombre d'items dans `_to_implement.md` (backlog actionnable),
   - nombre de gaps bloquants vs non-bloquants,
   - chemins des principaux livrables.

## Garde-fous

- Si `_codemap.md` n'est pas produit à l'étape 1 → arrêter et demander à
  l'utilisateur.
- Si le build échoue → afficher l'erreur Python, ne pas masquer.
- Ne jamais commit/push sans demande explicite — toute la sortie est
  locale.
- Argument optionnel `$ARGUMENTS` : si non vide, le passer comme
  périmètre aux agents (ex. `/doc-62304 packages/auth` pour limiter le
  scope au package auth).
