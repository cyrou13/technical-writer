Exécute le pipeline complet de génération documentaire IEC 62304 Classe A.
Sortie locale en Markdown dans `docs/`. Ne pousse rien à l'extérieur.

## Étapes

1. **Cartographier** le repo en lançant le sub-agent `code-archeologist`.
   Bloquant — attendre la fin avant l'étape 2.

2. **Rédiger en parallèle** (un seul message, plusieurs Agent calls) :
   - `requirements-writer` → `docs/items/SRS/`
   - `architecture-writer` → `docs/items/SDS/` (peut tourner en
     parallèle de `requirements-writer` puisque l'archi-writer remplit
     les `links.implements` à partir des SRS au moment du build, pas en
     écriture concurrente).
   - `test-evidence-collector` → `docs/items/TC/`

   Si tu préfères la séquence stricte (SRS d'abord pour que SDS et TC
   aient les IDs SRS pour leurs liens), lancer SRS seul d'abord, puis
   SDS+TC en parallèle.

3. **Build** : exécuter `python tools/build_docs.py`. Vérifier que les
   fichiers `docs/generated/{10_SRS,20_SDS,30_test_evidence,40_traceability}.md`
   et `coverage.json` ont été produits.

4. **Revue** : lancer le sub-agent `compliance-reviewer`. Il écrit
   `docs/generated/99_compliance_review.md`.

5. **Synthèse à l'utilisateur** : afficher en ≤ 10 lignes :
   - métriques de couverture extraites de `coverage.json`,
   - nombre de gaps bloquants vs non-bloquants,
   - chemins des principaux livrables.

## Garde-fous

- Si `_codemap.md` n'est pas produit à l'étape 1 → arrêter et demander à
  l'utilisateur.
- Si le build échoue → afficher l'erreur Python, ne pas masquer.
- Ne jamais commit/push sans demande explicite — toute la sortie est
  locale.
- Argument optionnel `$ARGUMENTS` : si non vide, le passer comme
  périmètre aux agents (ex. `/doc-62304 packages/auth` pour limiter le
  scope au package auth).
